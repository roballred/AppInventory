# State Application Inventory — Data Model

**Product:** State Application Inventory
**Version:** v1
**Last Updated:** 2026-03-27

This document describes what information the system stores and how the different pieces relate to each other. It is written for business and policy audiences — technical database detail is summarized rather than reproduced.

The data fields for application records are based on state IT portfolio policy minimum requirements (Washington State MGMT-01-01-S used as the example). Adopting states should map fields to their own equivalent policy standard.

---

## What the System Stores

The system manages six types of information:

| Record Type | What It Represents |
|-------------|-------------------|
| Agency | A state agency that owns application records |
| User | A person who accesses the system |
| Application | A single application in an agency's inventory |
| Audit Log | A full history of every change made to every application record |
| Notification | Alerts generated by business rules and sent to users |
| Certification | The annual certification submission record for each agency |

---

## Agency

Represents a state agency participating in the inventory program.

| Field | Description |
|-------|-------------|
| Name | Official agency name |
| Agency Number | Official agency identifier from the state IT authority |
| Identity Group | Links the agency to its users via the state identity provider |

---

## User

Represents a person who logs into the system. Users are authenticated through the state identity provider (SSO) — the system does not manage passwords.

| Field | Description |
|-------|-------------|
| Display Name | Person's name as shown in the app |
| Email | Contact email address |
| Agency | Which agency this user belongs to |
| Role | What this user is allowed to do (see Roles & Permissions document) |

**Roles:** Platform Admin, Agency Admin, Submitter, Viewer

Platform admins can see all agencies. All other roles are limited to their own agency's data only.

---

## Application

A single application record in an agency's inventory. The v1 field set focuses on the information most likely to change and most commonly incorrect.

**Required Fields**

| Field | Description |
|-------|-------------|
| Name | Name of the application |
| Lifecycle Status | Current stage: In development, In production, Retirement in progress, or Retired from inventory |
| Last Reviewed Date | When the record was last confirmed accurate — drives staleness detection |

**Optional Fields**

| Field | Description |
|-------|-------------|
| Description | Brief description of what the application does |
| Version | Current version number |
| Manufacturer / Vendor | Company that makes or supplies the application |
| Cloud Service Provider | Cloud platform if applicable |
| Operating System | OS the application runs on |
| OS Version | Specific OS version |
| Contract Number | Associated procurement contract |
| License Number | Software license identifier |
| In Service / Production Date | When the application went live |
| Retirement Date | Planned or actual retirement date |
| Unsupported Version | Yes/No — is this version no longer supported by the vendor? |
| Updatable | Yes/No — can this application be updated? |
| Aging Technology | Yes/No — flagged as aging or legacy |
| AI Enabled | Yes/No — does this application use AI? |
| Generative AI | Yes/No — does it specifically use generative AI? |

---

## Audit Log

Every change to every application record is recorded in the audit log. Each entry captures who made the change, when, what action was taken (created, updated, or retired), and exactly which fields changed along with the old and new values.

This log is permanent and supports staleness detection, compliance reporting, and accountability.

---

## Notification

Alerts generated automatically when business rules are triggered (e.g., a record becoming stale or a certification deadline approaching). Each notification records which rule triggered it, which user received it, how it was delivered (email or in-app), and when it was read.

---

## Certification

One record per agency per year. Tracks whether the agency's annual certification is not started, in progress, submitted, or approved. Records who submitted it and which platform admin approved it.

---

## How These Records Relate

- Each agency has many users and many applications
- Each application belongs to one agency
- Every change to an application creates an audit log entry linked to both the application and the user who made the change
- Notifications are linked to the agency and user who received them, and optionally to a specific application
- Each agency has one certification record per year
