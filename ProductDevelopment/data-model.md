# State Application Inventory — Data Model

**Product:** State Application Inventory
**Version:** v2
**Last Updated:** 2026-03-28

This document describes what information the system stores and how the different pieces relate to each other. It is written for business and policy audiences — technical database detail is summarized rather than reproduced.

The data fields for application records are based on state IT portfolio policy minimum requirements (Washington State MGMT-01-01-S used as the example). Adopting states should map fields to their own equivalent policy standard.

---

## What the System Stores

The system manages six types of information:

| Record Type | What It Represents |
|-------------|-------------------|
| Agency | A state agency that owns application records |
| User | A person who accesses the system |
| Application | A single application in an agency's inventory |
| Audit Log | A full history of every change made to every application record |
| Notification | Alerts generated by business rules and sent to users |
| Certification | The annual certification submission record for each agency |
| Work Queue Dismissal | A record of Jordan dismissing a queue item as already handled (CAP-09) |
| Review Assignment | A record of Maria assigning an application record for review (CAP-10) |
| Agency Data Quality Flag | A record of Derek flagging an agency for follow-up (CAP-08) |

**Computed views (no stored table):** The work queue (CAP-09), the Agency Risk Dashboard (CAP-10), and the Data Quality Monitoring screen (CAP-08) are all derived in real time from Application records and the business rules thresholds. Only the three action records above require persistent storage.

---

## Agency

Represents a state agency participating in the inventory program.

| Field | Description |
|-------|-------------|
| Name | Official agency name |
| Agency Number | Official agency identifier from the state IT authority |
| Identity Group | Reserved for future use — will link the agency to its users via the state identity provider's group claims for automatic agency assignment during sign-in. Not yet implemented. Manual role assignment is used in v1. |

---

## User

Represents a person who logs into the system. Users are authenticated through the state identity provider (SSO) — the system does not manage passwords.

| Field | Description |
|-------|-------------|
| Display Name | Person's name as shown in the app |
| Email | Contact email address |
| Agency | Which agency this user belongs to |
| Role | What this user is allowed to do (see Roles & Permissions document) |

**Roles:** Platform Admin, Agency Admin, Submitter, Viewer

Platform admins can see all agencies. All other roles are limited to their own agency's data only.

---

## Application

A single application record in an agency's inventory. The v1 field set focuses on the information most likely to change and most commonly incorrect.

**Required Fields**

| Field | Description |
|-------|-------------|
| Name | Name of the application |
| Lifecycle Status | Current stage: In development, In production, Retirement in progress, or Retired from inventory |
| Last Reviewed Date | When the record was last confirmed accurate — drives staleness detection |

**Optional Fields**

| Field | Description |
|-------|-------------|
| Description | Brief description of what the application does |
| Version | Current version number |
| Manufacturer / Vendor | Company that makes or supplies the application |
| Cloud Service Provider | Cloud platform if applicable |
| Operating System | OS the application runs on |
| OS Version | Specific OS version |
| Contract Number | Associated procurement contract |
| License Number | Software license identifier |
| In Service / Production Date | When the application went live |
| Retirement Date | Planned or actual retirement date |
| Unsupported Version | Yes/No — is this version no longer supported by the vendor? |
| Updatable | Yes/No — can this application be updated? |
| Aging Technology | Yes/No — flagged as aging or legacy |
| AI Enabled | Yes/No — does this application use AI? |
| Generative AI | Yes/No — does it specifically use generative AI? |
| Technical Owner Email | Email address of the technical owner — used by the notification engine to route stale record alerts. Required before CAP-04 (notifications) can route alerts to the correct person. |

**Risk Field Verification**

The five risk flags above (Unsupported Version, Updatable, Aging Technology, AI Enabled, Generative AI) are self-reported by agency staff. The system tracks whether any of these fields have ever been explicitly set after the record was created.

| Field | Description |
|-------|-------------|
| Risk Fields Last Verified | The date and time when a user last explicitly reviewed and saved any risk flag — distinct from record creation |

If this date is null, it means the risk flags have never been updated since the record was initially created. The Agency Risk Dashboard (CAP-10) displays an **Unverified** indicator on any risk item where the underlying record has never had its risk flags verified. This makes uncertainty visible to the agency admin rather than presenting unreviewed data as confirmed fact.

This does not block certification or hide the record from risk views. It is informational — Maria sees the risk and its confidence level, and can assign it to Jordan for verification.

---

## Audit Log

Every change to every application record is recorded in the audit log. Each entry captures who made the change, when, what action was taken (created, updated, or retired), and exactly which fields changed along with the old and new values.

This log is permanent and supports staleness detection, compliance reporting, and accountability.

---

## Notification

Alerts generated automatically when business rules are triggered (e.g., a record becoming stale or a certification deadline approaching). Each notification records which rule triggered it, which user received it, how it was delivered (email or in-app), and when it was read.

---

## Certification

One record per agency per year. Tracks whether the agency's annual certification is not started, in progress, submitted, or approved. Records who submitted it and which platform admin approved it.

---

## Work Queue Dismissal (CAP-09)

The work queue itself is a **computed view** — the system derives what Jordan needs to see by querying application records against the staleness thresholds, required field list, and risk verification status defined in the business rules. No separate "queue" table is needed.

The one piece of state the work queue must store is a **dismissal**: when Jordan dismisses an item as already handled, the system records that so the item does not immediately reappear on the next page load.

| Field | Description |
|-------|-------------|
| User | The submitter who dismissed the item |
| Application | The record that was dismissed |
| Reason | Why it appeared in the queue: stale, missing fields, or unverified risk |
| Dismissed At | When the dismissal was recorded |

**Dismissal expiry:** A dismissal is automatically cleared when the application record is updated (the Last Reviewed Date changes) or after 7 days, whichever comes first. This prevents Jordan from permanently hiding a record that genuinely needs attention.

---

## Review Assignment (CAP-10)

When Maria assigns a record for review from the Agency Risk Dashboard, the system creates an assignment record linking the application to the user who should review it.

| Field | Description |
|-------|-------------|
| Application | The record assigned for review |
| Assigned By | The agency admin who created the assignment |
| Assigned To | The user asked to review the record |
| Assigned At | When the assignment was created |
| Resolved At | When the user completed the review (null if still open) |
| Notes | Optional context Maria provides when assigning |

Open assignments are visible in the assignee's work queue as a higher-priority item than standard staleness alerts.

---

## Agency Data Quality Flag (CAP-08)

The Data Quality Monitoring screen is a **computed view** — stale record counts, missing field counts, and inconsistent value patterns are derived in real time from application records. No summary table is stored.

The one piece of state CAP-08 must store is a **follow-up flag**: when Derek flags an agency for follow-up, that flag and its context need to persist so Derek can track open issues across sessions.

| Field | Description |
|-------|-------------|
| Agency | The agency being flagged |
| Flagged By | The platform admin who created the flag |
| Flagged At | When the flag was set |
| Reason | Brief note on what needs follow-up |
| Resolved At | When Derek marked the issue resolved (null if still open) |

## Schema Evolution Policy

The data model will change over time as policy evolves — new risk flags, new classification fields, new reporting requirements. This section defines how those changes are governed so that existing agency data is never invalidated by a policy update.

### Field Tiers

Fields are classified by how they change — not by importance.

| Tier | Description | Examples |
|------|-------------|---------|
| 1 — Identity | Set at record creation. Do not change as policy evolves. | Application name, agency, created date, in-service date |
| 2 — Operational | Change because the application changes. Tracked in the audit log via normal edits. | Lifecycle status, version, vendor, OS, contract number |
| 3 — Policy | Added or modified as state IT policy evolves. Most likely to change between certification cycles. | Risk flags: AI Enabled, Generative AI, Unsupported Version, Aging Technology, and any future equivalents |

### Governance Rules

**1. New fields can be added at any time.**
Existing records receive a null value for the new field and are not invalidated. The audit log records the addition.

**2. New required fields have a one-cycle grace period.**
A field that becomes required is treated as optional for existing records until the next certification deadline. Agencies are notified at the start of the new cycle that the field must be completed before they can certify. New records created after the field is added must include it immediately.

**3. Fields are never removed — only deprecated.**
A deprecated field is hidden from the Add / Edit Application form but its data is preserved in the database and remains visible in the Application Detail view and audit history. This protects historical records and ensures the audit log remains interpretable.

### Technical Mechanism

Schema changes are applied using Drizzle Kit migration files. Each change is a numbered, versioned migration script committed to the repository. Migrations run automatically on deployment. No manual database changes are made outside of this process.

## How These Records Relate

- Each agency has many users and many applications
- Each application belongs to one agency
- Every change to an application creates an audit log entry linked to both the application and the user who made the change
- Notifications are linked to the agency and user who received them, and optionally to a specific application
- Each agency has one certification record per year
- Work Queue Dismissals are linked to a specific user and application; they expire automatically and are not shown in the audit history
- Review Assignments are linked to an application, the assigning admin, and the assigned user; open assignments surface in the assignee's work queue
- Agency Data Quality Flags are linked to an agency and the platform admin who created them; they persist until marked resolved
